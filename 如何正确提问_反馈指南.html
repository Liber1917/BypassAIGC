一、提问的智慧——核心原则

本指南基于 Eric S. Raymond 的经典文章 《How To Ask Questions The Smart Way》（《提问的智慧》）提炼而来。核心理念只有一句话：

尊重他人时间，证明你值得被帮助。 好的问题节省双方的时间，坏的问题浪费所有人的耐心。

在技术社区和团队协作中，提问不是"索取"，而是一种合作。你准备得越充分，你得到的答案就越快、越准确。

二、提问前，先做这四件事

① 自己搜索

在问别人之前，先尝试自己找到答案。这是一个"证明你用过脑"的基本门槛。

  • 用搜索引擎搜索报错信息
  • 翻阅 README / 使用文档 / FAQ
  • 在已有 GitHub Issues 中搜索是否有人问过相同问题
  • 查看程序日志（exe 同目录下的 logs/ 文件夹）

② 自己排查

  • 检查 .env 配置是否正确（API Key、端口等）
  • 重启程序，看看问题是否复现
  • 换个浏览器试试（排除缓存问题）
  • 看看 http://localhost:9800/docs 后端 API 是否正常

③ 准备好信息

把以下信息提前准备在手上，而不是等人问一句才补一句：

  • 程序版本（exe 文件名上的版本号/构建日期）
  • 操作系统版本（Win 10 / Win 11 / macOS）
  • 完整报错信息（截图或文本）
  • 复现步骤

④ 表明你做过了什么

在提问时明确写出你已经尝试过的操作，这能极大提高回答者帮助你的意愿。

示例："我已经搜索了 Issues，没有找到类似问题。我试过重启程序、重新配置 .env、换 Chrome 和 Edge 浏览器，但问题仍然存在。"

三、提问时，注意这五点

① 标题要清晰具体

不要用「救命！」「急！求助！」这类标题。好的标题让回答者一眼就知道你在问什么。

程序出错了，求大神帮忙
「论文生成」页面点击生成后无响应，浏览器 Console 报 net::ERR_CONNECTION_REFUSED

② 描述症状，而不是猜测

告诉别人 发生了什么，而不是你认为 为什么会这样。

是不是后端挂了？是不是 API Key 有问题？
点击「保存」按钮后，页面弹出「500 Internal Server Error」，程序窗口的日志显示：sqlalchemy.exc.OperationalError

③ 提供复现步骤

让别人能按你的描述一步一步重现问题。这是修复 Bug 最重要的信息。

# 复现步骤示例 1. 打开程序，访问 http://localhost:9800 2. 点击右上角「管理后台」 3. 输入管理员密码登录 4. 点击「用户管理」→「创建用户」 5. 填写用户名后点击「提交」 6. 页面报错："Failed to create user"

④ 附上日志和截图

一图胜千言，一段日志胜过十句描述。

  • 程序日志：exe 同目录下的 logs/ 文件夹
  • 浏览器日志：F12 → Console 面板 → 右键 → "Save as" 保存
  • 截图：直接拖放图片到 Issue / 聊天窗口

⑤ 问题解决后，说一声

在 Issue 或聊天中补充说明问题是如何解决的。这不仅帮助后来人，也是对所有帮助者的尊重。

示例："已解决！原因是 .env 中 ADMIN_PASSWORD 包含特殊字符需要转义。谢谢大家的帮助！"

四、好问题 vs 坏问题 对比

场景：程序启动报错

打不开，报错了，怎么办？
双击 AI学术写作助手.exe 后，窗口一闪就关闭了。exe 同目录下 logs/ 目录中的日志最后几行是：
ERROR:root:Failed to bind to port 9800: Address already in use
我已经检查过没有其他程序占用 9800 端口，但仍无法启动。操作系统是 Windows 11，exe 版本是今天（2026-05-19）构建的。

场景：功能异常

论文生成功能用不了
在我正在编辑的一篇约 5000 字的论文中，点击「生成摘要」按钮后，页面一直显示「正在生成...」，等了 5 分钟没有反应。我检查了浏览器 Console（附截图），没有报错信息。OPENAI_API_KEY 我已确认有效（在别的工具中可以正常使用），且网络可以访问 api.openai.com。

❌ 不要问这些问题

  • "谁有空？"—— 没人会主动认领
  • "急！在线等！"—— 着急不会让答案来得更快
  • "我的电脑有问题"—— 太模糊，无从下手
  • "代码太烂了"—— 不是在解决问题，是在发泄情绪
  • "能不能帮我做……"—— 请先展示你的努力

五、📋 Bug 反馈模板（复制即用）

当你在使用过程中遇到问题时，直接复制下面的模板，填写你的信息后发给开发者：

## 🐛 Bug 反馈 【问题描述】 （请用一两句话描述你遇到的问题） 【复现步骤】 1. 2. 3. 【预期结果】 （你认为应该发生什么？） 【实际结果】 （实际上发生了什么？） 【日志 / 截图】 （请粘贴日志内容或拖放截图到这里） 【程序版本】 （exe 版本号或构建日期） 【操作系统】 （Windows 10/11 / macOS 版本号） 【其他信息】 （网络环境、代理设置、特殊配置等）

六、📋 功能建议模板（复制即用）

## 💡 功能建议 【背景 / 场景】 （在什么场景下遇到这个需求？当前有什么痛点？） 【需求描述】 （你希望程序新增或改进什么功能？） 【验收标准】 - [ ] - [ ] - [ ] 【优先级】 （P0-核心急需 / P1-重要 / P2-值得做 / P3-锦上添花） 【参考示例】 （如果有同类产品的截图或链接，请附上）

七、🌐 在 GitHub 上提交 Issue

如果你有 GitHub 账号，直接在项目仓库提交 Issue 是最高效的方式：

  1. 打开项目地址：https://github.com/Liber1917/BypassAIGC
  2. 选择「缺陷报告 / Bug Report」或「功能建议 / Feature Request」
  3. 按表单提示填写信息即可（带 * 的为必填）
  4. 提交后开发者会收到通知，定期处理
提示：提交前记得先搜索已有 Issues，避免重复。提交后可以通过后续评论补充更多信息。

本文参考 Eric S. Raymond 的 How To Ask Questions The Smart Way（《提问的智慧》）编写，祝大家提出好问题，获得好答案。
